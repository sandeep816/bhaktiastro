## Purpose and scope

A BhaktiAstro golden fixture is a stable, provenance-backed, independently
verified reference dataset used to validate deterministic calculation output.
Its expected values come from declared evidence outside the BhaktiAstro output
under test, are reviewed under an explicit tolerance policy, and cannot be
silently changed to make a test pass.

A golden fixture is not merely:

- a convenient request or response example;
- a snapshot generated by the current implementation;
- copied runtime output, even when that output is deterministic;
- an undocumented expected value;
- a structural smoke fixture whose keys and value types happen to pass;
- a synthetic mock used to exercise orchestration; or
- a passing regression test without independent astronomical provenance.

This specification defines governance, classification, provenance, time and
location representation, schema/version policy, storage boundaries, future
fixture scopes, and review requirements. It creates no fixture data and makes
no astronomical value verified.

## Terminology and closed fixture classifications

Fixture classification describes why a dataset exists. Verification method is
recorded separately in provenance. The closed classification vocabulary is:

| Classification | Exact meaning |
| --- | --- |
| `structural` | Preserves keys, value types, nesting, or response shape. Values may be generated by current runtime and carry no accuracy claim. |
| `regression` | Preserves reviewed behavior associated with a defect or compatibility contract. It is not astronomically authoritative unless it separately qualifies as a new golden fixture. |
| `provisional_reference` | Contains externally sourced candidate values undergoing provenance, comparison, tolerance, or reviewer completion. It must not drive an enabled golden-value assertion. |
| `golden` | Meets every qualification rule below, has lifecycle status `approved`, `active`, or `superseded`, and carries independently reviewed expected values for its declared scope. |

`manual reference` and `external reference` are verification methods, not
additional fixture classifications. The distinction avoids a fixture being
simultaneously classified by purpose, storage format, and evidence source.
Test-vector categories such as `manual_astronomical` and `external_reference`
remain valid and may describe the human-readable record linked to a fixture.

No unspecified classification is accepted. Adding or renaming a classification
requires a schema and compatibility decision.

## Fixture lifecycle

The closed lifecycle-status vocabulary is:

| Status | Meaning |
| --- | --- |
| `proposed` | Scope and identity are recorded; source evidence may be absent. |
| `sourced` | Required source records and versions are present but no independent comparison is complete. |
| `independently_checked` | Expected values have been compared to at least one independent source and differences are recorded. |
| `in_review` | A named reviewer is evaluating provenance, comparison, tolerance, and unresolved discrepancies. |
| `approved` | Review is accepted and every golden qualification criterion is satisfied, but repository test activation may still be pending. |
| `active` | The approved fixture is stored in canonical locations and enforced by enabled tests for its declared scope. |
| `superseded` | A named later fixture version replaces it; the record remains immutable historical evidence. |
| `rejected` | Evidence, provenance, scope, or comparison failed review; the reason is retained. |

Normal progression is `proposed` to `sourced` to `independently_checked` to
`in_review` to `approved` to `active`. Review may return a fixture to an
earlier non-active state or reject it. Only a separately reviewed replacement
may supersede an active golden fixture.

## Golden qualification criteria

A fixture may use classification `golden` only when all of the following are
present and valid:

1. a unique stable fixture identifier and exact schema ID/version;
2. one or more explicitly declared calculation scopes;
3. city, country, coordinates, and the location assumptions needed by those
   scopes;
4. an explicit local-time representation, IANA timezone identifier, UTC
   instant where one exists, UTC offset, calendar, and DST/fold treatment;
5. all calculation configuration that can affect expected output, including
   ayanamsa, reference point, ephemeris mode/version, and house convention
   where relevant;
6. finite, ordered expected outputs limited to the declared scope;
7. a field- or output-family-specific comparison rule and justified tolerance;
8. at least one source independent of the BhaktiAstro output under test,
   including source type, citation, version, configuration, and access or
   reference date;
9. a second independent source where practical, or a reviewer-approved written
   explanation of why one authoritative primary source is sufficient;
10. the comparison method and all observed differences, including values that
    agree exactly;
11. reviewer identity, verification date, approval state, and absence of an
    unresolved difference outside the accepted tolerance;
12. human-readable provenance linked to the machine-readable fixture;
13. deterministic field and collection order; and
14. no expected value copied solely from BhaktiAstro runtime output.

The fixture need not validate every possible calculation. A smaller fixture
with complete evidence for one declared scope is preferable to a broad fixture
with undocumented or unverified fields.

## Fixture schema and exact top-level order

The future canonical machine-readable envelope has:

```text
GOLDEN_FIXTURE_SCHEMA_ID = "bhaktiastro.golden-fixture"
GOLDEN_FIXTURE_SCHEMA_VERSION = "1.0"
```

Its exact top-level field order is:

| Position | Field | Contract |
| ---: | --- | --- |
| 1 | `schema_id` | Exact schema identifier above |
| 2 | `schema_version` | Exact supported version above |
| 3 | `fixture_id` | Stable identifier defined below |
| 4 | `classification` | One closed fixture classification |
| 5 | `lifecycle_status` | One closed lifecycle status |
| 6 | `scope` | Non-empty ordered unique calculation-scope list |
| 7 | `location` | Exact location and timezone context |
| 8 | `time` | Exact local/UTC/calendar/DST context |
| 9 | `calculation_config` | Ordered calculation-affecting configuration |
| 10 | `expected` | Ordered outputs limited to declared scope |
| 11 | `tolerances` | Ordered per-field or per-family comparison policies |
| 12 | `provenance` | Non-empty ordered independent source records |
| 13 | `review` | Comparison and approval record |
| 14 | `supersedes` | Empty string or exact prior fixture ID |
| 15 | `notes` | Ordered technical notes; no interpretation |

Unknown fields, missing required fields, wrong order, duplicate scopes,
duplicate IDs, unsupported vocabulary, non-finite values, and inconsistent
cross-fields are invalid. Schema `1.0` accepts no extension bucket and performs
no silent defaulting or migration.

The initial closed calculation-scope vocabulary is:

```text
julian_day
ayanamsa
planetary_positions
panchang_elements
panchang_boundaries
rise_set
kundali_base
timezone_validation
```

A fixture declares only the scopes for which it has complete expected values,
provenance, and tolerances. Adding a scope token is a schema-vocabulary review.

### Exact nested record shapes

`calculation_config` always has this exact order; irrelevant string values are
empty and irrelevant ordered collections are empty rather than omitted:

```text
ephemeris_name
ephemeris_version
ayanamsa
time_scale
coordinate_frame
node_mode
planet_set
house_system
rise_set_model
calendar
enabled_sections
```

`expected` is a non-empty ordered list. Each expected-value record has exact
order:

```text
scope
field_path
value
source_ids
notes
```

`scope` is one declared fixture scope. `field_path` is a non-empty ordered list
of machine field-name strings and non-negative list indexes locating a value in
the owning calculation result. `value` is a finite JSON-safe scalar or ordered
container. `source_ids` is a non-empty ordered unique list resolving to
provenance records. One field path appears at most once.

`tolerances` has exactly one record for each expected record in the same order.
Each record has exact order:

```text
field_path
comparison_mode
tolerance_value
unit
source_precision
justification
```

`comparison_mode` is `exact`, `absolute`, `circular_degrees`, or `pending`.
`tolerance_value` is null for `exact` and `pending`, and otherwise a finite
non-negative number. `unit` is `none`, `degrees`, `seconds`, or `days` and must
match the comparison. `source_precision` and `justification` are non-empty for
an approved golden tolerance. A `pending` record prevents `golden`, `approved`,
and `active` states.

Each `provenance` item has the exact ten-field order defined in the provenance
section. `review` has exact order:

```text
reviewer
review_date
comparison_method
observed_differences
single_source_exception
unresolved_issues
decision
```

`observed_differences` is ordered by expected-field order. The review decision
is `pending`, `approved`, or `rejected` and must agree with lifecycle status.
`single_source_exception` is empty when two or more independent sources exist;
otherwise it contains the reviewer-approved justification. `notes` is an
ordered list of technical strings.

## Stable fixture identifiers

Fixture IDs use:

```text
GF-<LOCATION>-<YYYYMMDD>-<SCOPE>-V<N>
```

- `GF` is literal.
- `LOCATION` is an uppercase ASCII city/location token with optional underscore
  components and is not silently derived from coordinates.
- `YYYYMMDD` is the fixture's local civil date. One multi-instant transition
  study uses separate fixture IDs for each valid or invalid local-time case.
- `SCOPE` is an uppercase ASCII summary token for the declared scope set.
- `V<N>` is a positive fixture revision beginning at `V1`.

The exact pattern is:

```text
^GF-[A-Z0-9]+(?:_[A-Z0-9]+)*-[0-9]{8}-[A-Z0-9]+(?:_[A-Z0-9]+)*-V[1-9][0-9]*$
```

IDs are repository-global and never reused. A change to expected values,
calculation configuration, time interpretation, tolerance semantics, source
meaning, or declared scope creates a new `V<N>` ID and links `supersedes`.
Correcting prose that cannot affect comparison may retain the ID only when the
review record documents the correction. Superseded and rejected IDs remain
reserved permanently.

## Location contract

`location` records, in exact order:

1. `city`;
2. `country_code` using an explicitly documented standard;
3. `latitude_degrees`, north positive;
4. `longitude_degrees`, east positive;
5. `elevation_meters`, or null when the declared calculation does not use it;
6. `timezone_id`, an IANA timezone database identifier;
7. `tzdata_version`; and
8. `coordinate_source`.

Coordinates are finite and remain within geographic ranges. Coordinate
precision and datum/source are recorded; a city-centre label never substitutes
for coordinates. Elevation is required when the calculation or reference
source uses it and must otherwise be explicitly null rather than guessed.

An IANA timezone ID is provenance context even though current Panchang and
Kundali APIs accept a numeric `timezone_offset`. Future fixture adapters may
pass only fields supported by the owning API; this specification does not add
timezone-ID input to runtime.

## Time and calendar contract

`time` distinguishes `valid_instant` from `invalid_local_time` cases.

For a `valid_instant`, it records in exact order:

1. `case_type` = `valid_instant`;
2. `local_date` as `YYYY-MM-DD`;
3. `local_datetime` as timezone-aware ISO 8601 including numeric offset;
4. `utc_datetime` as ISO 8601 with `Z`;
5. `utc_offset_seconds` as the exact signed integer offset;
6. `dst_status` as `standard`, `daylight`, `not_observed`, or `historical`;
7. `fold` as `0`, `1`, or null when the local time is unambiguous; and
8. `calendar`, explicitly naming the civil-calendar assumption.

The local timestamp, offset, fold, timezone ID, and UTC instant must represent
one consistent instant. No environment timezone or current system tzdata is
silently used.

For an `invalid_local_time` transition case, the record contains the local
wall-clock timestamp without an invented offset, `utc_datetime` is null,
`utc_offset_seconds` is null, `fold` is null, and expected output is a precise
validation rejection under `timezone_validation`. A nonexistent time never
receives a fabricated UTC equivalent. Repeated local times are two distinct
valid fixtures with explicit folds and UTC instants.

## DST validation policy

Every DST-sensitive fixture records:

- the IANA timezone identifier and tzdata version;
- expected UTC offset at the exact instant;
- expected DST status when represented by the source;
- transition provenance and the rule/version used to identify the transition;
- local and UTC instants on both sides when transition behavior is in scope;
- fold for repeated times;
- explicit rejection for nonexistent local times; and
- any historical timezone limitation in source or runtime support.

Timezone offset equality is exact at the integer-second level. A time-of-event
tolerance cannot excuse an incorrect timezone offset, fold, or UTC conversion.
When tzdata changes, the fixture is reviewed against the pinned source version;
it is not regenerated silently from the new environment.

## Calculation configuration

`calculation_config` records only declared deterministic configuration in the
exact nested order above. Its fields cover:

- ayanamsa mode and spelling;
- ephemeris source/engine and version;
- geographic or topocentric assumptions;
- atmospheric/refraction assumptions for rise/set values;
- node mode and planet set;
- Rashi, Nakshatra, house, and reference-point conventions;
- calendar and time-scale assumptions; and
- enabled optional Kundali sections.

Absent configuration is not inferred from current defaults. Irrelevant values
use the explicit empty representation required above and are never omitted.
Task 15.1 defines this content without creating a serializer or machine
fixture.

## Permitted astronomical validation scope

Sprint 15 fixtures may validate independently sourced values for:

- Julian Day and explicit local-to-UTC conversion;
- ayanamsa;
- tropical or sidereal planetary longitudes and derived Rashi identity;
- Tithi, Nakshatra, Yoga, Karana, and their declared boundaries;
- sunrise, sunset, moonrise, and moonset, including no-event cases;
- sidereal ascendant and Rashi;
- base whole-sign house structure; and
- timezone/DST conversion behavior around selected cases.

Each fixture declares a subset. Sunrise does not imply sunset; Panchang element
identity does not imply its boundary time; a planetary longitude does not imply
all derived Kundali metadata. Interpretations, predictions, remedies, or
judgments are never fixture scopes.

## Tolerance and comparison policy

Every expected field is paired with one comparison mode:

- `exact`: strings, booleans, categorical identifiers, indexes, integer counts,
  list membership/order, and timezone offset seconds must match exactly;
- `absolute`: a reviewed numeric or elapsed-time tolerance with explicit unit;
- `circular_degrees`: shortest angular difference across `0`/`360` with an
  explicit reviewed degree tolerance; or
- `pending`: candidate data that cannot qualify as golden or activate an
  exact-value test.

No universal floating-point or event-time tolerance is selected. The current
Validation Plan's `0.001`-day Julian check, planetary-longitude MVP/stable
targets, Panchang-boundary targets, and rise/set targets remain **pending
review targets**, not approved golden tolerances. A future fixture task must
justify its tolerance from reference precision, source agreement, calculation
precision, and reviewer evidence.

Comparisons use unrounded canonical numeric values when available. Display
rounding is separately documented and never widens a tolerance. A tolerance
cannot convert a category across a Rashi, Nakshatra, Tithi, Yoga, Karana, house,
date, or DST boundary. Circular longitude comparison uses shortest angular
distance so values near `0` and `360` are not compared linearly. All numbers
must be finite; booleans are not numbers.

## Provenance contract

Each ordered provenance source records:

1. `source_id` unique within the fixture;
2. `source_type`;
3. publication, software, standard, or service name;
4. publisher or owner;
5. exact edition/build/database/source version;
6. source URL or bibliographic citation;
7. access or reference timestamp;
8. calculation settings and assumptions used at that source;
9. retained evidence reference, when legally and operationally appropriate;
   and
10. technical notes and known limitations.

The closed source-type vocabulary is:

```text
authoritative_ephemeris
independent_reference_software
published_astronomical_table
trusted_public_standard
independent_manual_calculation
```

Acceptable sources include authoritative published ephemerides, independently
configured reference software, manually checked astronomical tables, trusted
public timezone/calendar standards, and a repository-owned calculation that
has itself been reviewed against an independent source. BhaktiAstro output
alone is never independent evidence.

Source URLs do not replace version and settings. Software screenshots may be
retained later as supporting evidence but are neither required nor sufficient
alone; the source version, input settings, and transcribed comparison must be
machine/audit readable. Copyright-restricted evidence is cited rather than
copied. Task 15.1 performs no web access, scraping, or screenshot capture.

## Future Kundali fixture structure

A future `kundali_base` fixture uses synthetic or historical/public input and
contains no unnecessary personal identifying information. The input identity
is a stable synthetic `subject_id`; personal name, email, address, account ID,
or private biography is excluded.

The fixture envelope records:

- subject ID;
- birth local date/time and consistent UTC instant;
- location and timezone context;
- calculation configuration, including ayanamsa and enabled options;
- provenance for each expected output family; and
- schema/version and fixture-version fields.

The ordered `expected` Kundali payload is:

1. `lagna`, including sidereal longitude, Rashi identity, and degree where
   independently verified;
2. `planets`, ordered `sun`, `moon`, `mars`, `mercury`, `jupiter`, `venus`,
   `saturn`, `rahu`, `ketu`, with only declared and sourced fields;
3. `houses`, ordered one through twelve when whole-sign house structure is in
   scope; and
4. `derived`, containing only separately declared optional fields whose owning
   runtime contracts and reference evidence are identified.

Rashi, house, retrograde, dignity, aspect, Varga, strength, Ashtakavarga,
special-Lagna, and Prediction fields are not automatically included merely
because runtime can emit them. Expected collection order must match the owning
contract. Each numeric family has its own reviewed tolerance. No actual
Kundali fixture values are created by Task 15.1.

## Relationship to test vectors and runtime fixtures

- This specification owns cross-domain golden-fixture governance.
- `docs/test-vectors/` owns human-readable reviewed cases and provenance.
- Machine JSON files provide deterministic test input/expected data but do not
  prove their own accuracy.
- Pytest fixtures/helpers load or construct test data; they do not automatically
  create canonical fixtures.
- Structural fixtures validate shape only.
- Regression fixtures preserve a documented behavior or defect contract.
- Manual-reference and external-reference test-vector categories describe how
  a human record was verified.
- A golden fixture links all applicable layers by stable ID and retains one
  authoritative lifecycle state.

Generated runtime output may help compare structure, but it is never copied
back as expected golden data without independent source review.

## Validation and activation workflow

For each future fixture:

1. assign a unique ID, `proposed` status, and narrow scope;
2. select location/time/configuration without deriving hidden defaults;
3. record independent source names, versions, settings, and citations;
4. transcribe candidate values as `provisional_reference` data;
5. compare every expected field using a proposed per-field policy;
6. record all differences and source disagreements;
7. obtain named review of provenance, scope, and tolerances;
8. mark the new fixture `golden` and `approved` only when qualification is
   complete;
9. store linked human and machine records;
10. add schema, provenance, calculation, and regression tests; and
11. mark `active` only after the focused and required regression suites pass.

Approval is never inferred from a passing structural test or a clean runtime
comparison.

## Failure and disagreement policy

- If current code disagrees with an active golden fixture, the test fails. Do
  not edit the fixture or widen tolerance. Investigate code, configuration,
  source version, time conversion, and fixture evidence through a focused task.
- If trusted sources disagree, retain every observation, keep the candidate in
  `in_review`, and do not select the convenient value. A named reviewer must
  resolve or reject the scope before approval.
- If tzdata changes, compare old and new rules against the recorded source
  version. A changed expected instant requires a new fixture revision and
  supersession review.
- If an ephemeris version changes, record both versions and create a new fixture
  revision when expected values or tolerance semantics change.
- If tolerance is exceeded, the fixture cannot be approved or the active test
  remains failing until the source of the discrepancy is resolved.
- If provenance is incomplete, classification cannot advance beyond
  `provisional_reference` and no golden assertion is enabled.
- If golden data is wrong, preserve it as superseded or rejected evidence,
  create a new reviewed fixture version, and never rewrite history silently.

Unexpected exceptions and malformed fixture data propagate through future
strict validators; no partial pass or fallback source is allowed.

## Schema and fixture compatibility

Schema ID, schema version, exact fields/order, nested shapes, vocabularies,
identifier format, and validation behavior follow
[SPEC-API-STABILITY-001](API-STABILITY.md).

- Schema `1.0` is strict; adding, removing, renaming, reordering, or retyping a
  field, or changing a closed vocabulary, is breaking for an active fixture.
- A minor schema version is allowed only when a future reader explicitly
  supports the extension while preserving all old `1.0` behavior.
- An incompatible change requires a new major schema version and coexistence
  plan.
- Unsupported schema IDs and versions are rejected; there is no nearest-version
  fallback, silent migration, upgrade, downgrade, or unknown-field repair.
- Fixture revision and fixture-schema version are independent. A fixture value
  correction increments its `V<N>` ID; a schema change does not silently
  rewrite existing fixture IDs or files.
